## Bug Fixes

- **Keycloak AppHost config** (`c03d396`) — Removed the `IsRunMode` conditional block that called
  `.WithDataVolume()` and a duplicate `.WithRealmImport()` on the Keycloak resource. The duplicate
  import caused realm duplication errors on startup; the data volume caused stale realm files to
  persist across sessions. Keycloak lifetime changed to `Session` and `--verbose` flag added.

- **Realm filename case mismatch** (`a8ac169`) — Renamed `toten-realm.json` → `ToTen-realm.json`.
  Keycloak's `DirImportProvider` enforces that the filename matches the `"realm"` key value
  exactly (case-sensitive). Forced via `git mv -f` to bypass macOS case-insensitive filesystem.
  Also erased drifted EF migration chain: `InitialCreate` had been rewritten after its first
  apply, leaving the database with an orphaned `Items` table. All migrations deleted; fresh
  single-migration `InitialCreate` generated from the current 15-table model snapshot.

- **Keycloak health check stall** (`760b2f7`) — `AddKeycloak` auto-registers a
  `HealthCheckAnnotation` against the HTTPS management port (9000). That port uses an
  Aspire-generated self-signed cert (`CN=localhost`) rejected by the AppHost's `HttpClient`,
  so `WaitFor(keycloak)` stalled indefinitely. Removed the annotation via
  `ResourceAnnotationCollection.Remove` and replaced with
  `WithHttpHealthCheck("/realms/master")` on HTTP port 8080, which returns 200 once Keycloak
  has finished startup and realm initialisation.

- **PostGIS extension missing** (`a5cf0be`) — The default `postgres:17.6` image has no PostGIS
  shared libraries, so the EF migration's `CREATE EXTENSION IF NOT EXISTS postgis` failed.
  Switched `RunAsContainer` to `postgis/postgis:17-3.4` (PostgreSQL 17 + PostGIS 3.4). Tag
  `17-3.5` has no `linux/arm64` manifest on Docker Hub; `17-3.4` is the highest available
  arm64 tag.

- **Minimal API DI binding error** (`e50ee0f`) — `IEventPublisher` injected into a Minimal API
  endpoint handler was being read from the request body by the framework (not allowed in
  Minimal APIs). Added `[FromServices]` attribute to explicitly bind from DI container.

## Implementation Notes

### EF Core Migration Drift Diagnosis
When `dotnet ef` reports `relation "X" does not exist`, connect directly to the running Postgres
container and compare `\dt` output against `__EFMigrationsHistory`. A mismatch means a migration
was rewritten after apply. Recovery: delete all migration files and the snapshot, drop/recreate
the data volume, regenerate with `dotnet ef migrations add InitialCreate --output-dir Data/Migrations`.

### Aspire Keycloak Health Check Internals
`Aspire.Hosting.Keycloak` (13.x) wires a `HealthCheckAnnotation` against the `management`
endpoint (HTTPS/9000) using a self-signed cert that the AppHost process does not trust.
`ResourceAnnotationCollection` is `IList<IResourceAnnotation>` — use `OfType<HealthCheckAnnotation>().ToList()`
then `Remove(item)` (no `RemoveAll` method exists). Substitute with `/realms/master` on the
`http` endpoint.

### Docker Volume / Image Lifecycle
`ContainerLifetime.Session` does **not** clean up named Docker volumes — volumes are independent
of container lifecycle. Always pair `Session` lifetime with no `WithDataVolume()` call if the
intent is ephemeral state. When switching a container image, remove the existing volume explicitly;
data directories from different base images are not always compatible.

### arm64 Image Availability
Before pinning a Docker image tag in `WithImageTag`, verify the manifest includes
`linux/arm64/v8` — `docker pull <image>:<tag>` will exit non-zero with a manifest error on
Apple Silicon if the platform is absent. For PostGIS, `17-3.4` is the current arm64 ceiling.

## Implementation Notes

### Terraform Bootstrap Sequence (Chicken-and-Egg)

The `keycloak` module depends on an ACR image that doesn't exist until built and pushed, but
ACR doesn't exist until Terraform provisions it. Resolution: two-pass `-target` apply.
First pass: `terraform apply -target=module.registry -target=module.container_apps ...`
(all non-Keycloak resources). Then build and push the custom Keycloak image to ACR.
Second pass: `terraform apply` (full apply, provisions Keycloak ACA container app).

### SignalR Free Tier Limits

`Free_F1` hard limits: 20 concurrent WebSocket connections, 20,000 messages/day.
`AddAzureSignalR()` is skipped when `SignalR:ConnectionString` is empty — the hub remains
fully functional as self-hosted. Upgrade trigger: >15 concurrent connections or >16K
messages/day (leaves <20% headroom before hard rejection). See
`docs/adal/phase-3-cost-analysis.md` for full upgrade threshold matrix.

### PostgreSQL Burstable CPU Credit Model

`B_Standard_B2ms` accumulates credits when CPU < baseline; depletes when CPU > baseline.
Sustained workloads >20 minutes at 100% CPU will throttle to ~50% baseline. Monitor
`sql_cpu_credits_remaining` metric; set an alert at 20 credits. PostGIS spatial aggregation
queries are CPU-intensive but short-lived — credit model is adequate for portfolio traffic.

### Key Vault Emulator HTTPS

`azure-keyvault-emulator` listens on HTTPS only. The `GetEndpoint("vault")` call in AppHost
returns the HTTPS URL. Applications must trust the emulator's self-signed cert or disable
certificate validation in development (`ASPNETCORE_Kestrel__Certificates__Default__*` or
`HttpClientHandler.ServerCertificateCustomValidationCallback`). Aspire dashboard env var
inspection confirms `KeyVault__Uri` resolves before first request.

## Implementation Notes

### Dockerfile Build Context Must Be Repo Root

`docker build -f docker/api/Dockerfile .` must be run from the repository root, not from
`docker/api/`. The `COPY . .` instruction in the build stage copies the full source tree;
`dotnet publish` then resolves `<ProjectReference>` paths relative to the copied tree. If the
context is the Dockerfile's own directory, the `src/ToTen.Contracts/` and
`src/ToTen.ServiceDefaults/` directories are absent and the build fails with
`MSBUILD : error MSB1003`. The existing `.dockerignore` at the repo root already excludes
`terraform/`, `docs/`, `.adal/`, and `CHANGELOG/`, keeping the build context tight.

### Terraform Plan/Apply Split: Binary Plan File

`terraform plan -out=tfplan` writes a binary plan file that encodes all resolved variable
values and the exact diff. `terraform apply tfplan` executes that exact plan without
re-evaluating variables, making the apply deterministic and preventing drift between what
was reviewed and what was applied. The binary plan is never committed — it is produced and
consumed within the same `terraform` job. Sensitive variable values (passwords) are not
readable from the binary plan file in plain text.

### ACA Bootstrap Order for Api/Worker Images

On the first `terraform apply`, the `apps` module requires `api_image` and `worker_image`
to already exist in ACR. These images don't exist until pushed by the `docker-build-push`
job in the same pipeline run. The pipeline enforces the correct order:
`docker-build-push` (push images) → `terraform` job (apply with image URIs as `-var`
inputs). For a net-new environment with an empty ACR, the first `docker-build-push` run
must complete before `terraform apply` targets the `apps` module. If Terraform fails on
first apply due to image pull errors, re-run the workflow after images are confirmed in ACR.

### ZAP `fail_action: false` Rationale

The ZAP baseline scan runs passive checks plus active scanning (`-a` flag) against the
live ACA endpoint. On first scan, false positives from standard security headers (e.g.,
missing `X-Frame-Options` on a JSON-only API) will appear. `fail_action: false` prevents
these from blocking main branch deployments until `.zap/rules.tsv` is calibrated with
deliberate suppressions. Flip to `fail_action: true` once a clean baseline with known
suppressions is established. ZAP findings are reported as GitHub Issues (requires
`issues: write` permission set at the workflow level).

## Implementation Notes

### Aspire EF Core DI Override Pattern

`builder.AddNpgsqlDbContext<TContext>("name")` registers three services: `TContext`,
`DbContextOptions<TContext>`, `IDbContextFactory<TContext>`, and a hidden fourth:
`IDbContextOptionsConfiguration<TContext>`. This last registration applies Aspire's
connection-string validation at configuration-bind time. Removing the first three in
`WebApplicationFactory.ConfigureServices` is insufficient — the hook persists and fires when
DI resolves any `IOptions<DbContextOptions<TContext>>`. The complete override requires
`services.RemoveAll<IDbContextOptionsConfiguration<TContext>>()` (needs
`using Microsoft.EntityFrameworkCore.Infrastructure`) plus injecting `ConnectionStrings:<name>`
into `IConfiguration` via `AddInMemoryCollection`.

### Derived Factory Auth Override Pattern

`WebApplicationFactory<T>.WithWebHostBuilder(...)` returns a derived factory that inherits
all `ConfigureWebHost` registrations from the parent. If the parent already registered an
authentication scheme, the derived factory must NOT call `AddScheme` again — it throws
`InvalidOperationException: Scheme already exists`. Use `services.PostConfigure<TOptions>("SchemeName", opts => { ... })`
instead: it modifies the named options in-place without touching scheme registration.

### Robot Framework CI Secret

`ROBOT_API_KEY` must be added to GitHub repository secrets before the `robot-tests` job
can authenticate against the live ACA endpoint. This is a runtime bearer token obtained
from Keycloak after first successful ACA deploy — it is not a build secret and does not
appear in Terraform outputs.
