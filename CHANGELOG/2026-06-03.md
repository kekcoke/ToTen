## Features

### Free-Tier Infrastructure Profile (Terraform)

- **`terraform/modules/postgres/variables.tf`** — `sku_name` default `B_Standard_B1ms` (was `GP_Standard_D2s_v3`)

- **`terraform/modules/service-bus/variables.tf`** — `sku` default `Basic` (was `Standard`)

- **`terraform/variables.tf`** — root-level `postgres_sku` and `service_bus_sku` defaults updated to match module defaults

- **`terraform/modules/apps/main.tf`** — API container: `min_replicas` 1→0 (HTTP ingress triggers
  implicit ACA HTTP scale rule), `cpu` 0.5→0.25, `memory` 1Gi→0.5Gi. Worker container:
  `min_replicas` 1→0, added KEDA `custom_scale_rule` block (`azure-servicebus`,
  `queueName=ToTen-Worker-Queue`, `messageCount=5`); authenticates via existing `servicebus-conn`
  secret — no new module variables needed.

- **`terraform/modules/keycloak/main.tf`** — `min_replicas` 1→0. No explicit HTTP scale rule needed;
  ACA adds one implicitly for any container app with an `ingress` block.

Deploy with `./scripts/toten.sh adjust --profile free-tier`.
Type `confirm-destroy` at the Service Bus SKU prompt (Azure does not support in-place SKU changes;
the namespace is destroyed and recreated, all three queues included).
Projected rest-state cost: ~$18–20/month (from ~$190–215/month). Cold-start times: API ~5–10s,
Keycloak ~30–60s (JVM + realm import), Worker ~5s after KEDA trigger.

## Bug Fixes

### Provision Script Hardening (`scripts/toten.sh`)

- **smoke-tests command missing (CRITICAL)** — `./scripts/toten.sh smoke-tests` was documented in
  `CLAUDE.md` and `docs/deployment-cookbook.md` but was never wired into the dispatcher. Running
  it printed "Unknown option" and exited 1. Added `cmd_smoke_tests()` that loads
  `terraform/outputs.env` and calls `_run_smoke_tests` with the saved API and Keycloak FQDNs.

- **Bootstrap RBAC scope fails on clean subscription (BUG)** — The `User Access Administrator`
  role assignment targeted `toten-prod-rg`, which does not exist before the first `provision` run.
  Added an existence check via `az group show`; falls back to subscription scope with a warning
  when the resource group is absent.

- **Stale `usage()` text (BUG)** — The free-tier profile description still said "ACA scale-to-zero
  requires separate module changes; see docs/infra-free-tier-downgrade.md §2.3" after those module
  changes were applied. Updated to: "Minimal SKUs — B_Standard_B1ms Postgres, Basic Service Bus,
  scale-to-zero on all apps".

- **Teardown: no Key Vault purge warning (CRITICAL)** — `terraform/modules/key-vault/main.tf`
  sets `purge_protection_enabled = true` and `soft_delete_retention_days = 90`. After
  `terraform destroy`, the Key Vault enters soft-delete and its name is reserved for 90 days —
  re-provisioning with the same name fails until the retention period expires. No warning existed
  in the script or the cookbook. Added a three-line warning block after the `destroy` confirmation
  gate in `cmd_teardown`.

- **App rollback: only API image verified (BUG)** — `rollback --app` checked `api/toten-api` in
  ACR but not `worker/toten-worker`. A partial CI run (e.g., failed mid-job) can push the API
  image but not the Worker image for a given SHA. The script would update `secrets.tfvars` and
  run `terraform apply`, which then fails during the Worker container app update trying to pull
  a non-existent image. Added a symmetric ACR tag check for `worker/toten-worker` immediately
  after the API check; fails fast with a clear message before mutating any state.

### CI/CD Workflow (`azure-dev.yml`) *(committed in prior session on this branch)*

- **Robot Framework path typos** — `tests/tralalala.AcceptanceTests/requirements.txt` and
  `tests/tralalla.AcceptanceTests/` corrected to `tests/ToTen.AcceptanceTests/`.

- **JMeter cache-hit expression** — `steps:cache-jmeter.outputs.cache-hit` (colon syntax) changed
  to `steps.cache-jmeter.outputs.cache-hit` (dot syntax, required by GitHub Actions expression
  evaluator). The colon form silently evaluated to `''`, bypassing the cache-hit gate and
  re-downloading JMeter on every run.

- **Stale JMeter report artifact** — Added `rm -rf jmeter-report/` before the JMeter run step to
  prevent stale HTML report files from prior runs from being included in the upload artifact.

## Operational Notes

### Free-Tier Deployment Cookbook (`docs/deployment-cookbook.md` — local, gitignored)

14 error scenarios documented (E1–E14) covering the full `adjust` / `provision` / `rollback` /
`teardown` lifecycle. Three scenarios added this session:

- **E12 — Key Vault purge-protected name conflict after teardown**: `purge_protection_enabled=true`
  + 90-day soft-delete means re-provisioning fails for 90 days. Remediation: `az keyvault recover`
  (reactivates the same vault), or rename the KV suffix in Terraform before re-applying.

- **E13 — App rollback with schema-ahead database**: `rollback --app` re-pins the container image
  but does not roll back the EF Core schema. Older code can fail at startup if the DB is ahead.
  Remediation: `dotnet ef database update <MigrationName>` with a temporary Postgres firewall
  rule (see E7 procedure).

- **E14 — App rollback without `--profile` reverts infrastructure SKUs**: `PROFILE` defaults to
  `prod`. Running `rollback --app --sha <sha>` without `--profile free-tier` on a free-tier
  environment applies `prod.tfvars`, triggering Service Bus destroy+recreate and Postgres resize.

### KEDA Scale Rule Timing

Default KEDA poll interval for `azure-servicebus` is 30s. First message to a 0-replica Worker
can take up to ~35s before processing begins. `messageCount=5` was chosen to avoid spurious
scale-up on transient queue spikes while still providing timely wake-up under normal load.
To test: send ≥5 messages to `ToTen-Worker-Queue`; Worker should show 1 replica within ~35s.

### Bootstrap on Clean Subscription

When running `./scripts/toten.sh bootstrap` on a subscription where `toten-prod-rg` does not
yet exist (i.e., before first `provision`), the `User Access Administrator` role is assigned at
subscription scope rather than RG scope. After the first `provision` creates the resource group,
re-running `bootstrap` is idempotent and will not duplicate the role assignment — it checks for
existing assignments before creating new ones.
